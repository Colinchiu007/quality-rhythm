# Bug/问题反思复盘

## 什么时候用

每次你在测试项目功能或实际使用中**发现一个问题或 bug** 时触发。不是给线上故障用的，是给**开发自测阶段**的快速复盘。

目标不是「修这个 bug」——修 bug 是顺便的事。目标是**找到系统漏洞，堵上它，让同类问题不再出现**。

---

## 复盘框架

### 第一步：描述问题

一句话说清楚发生了什么，不要带推测。

```
现象: <实际发生了什么>
预期: <应该发生什么>
复现: <怎么复现的，必填>
```

### 第二步：根因定位（5 Whys）

逐层追问，直到找到**可操作的根本原因**，而不是停在表面。

```
问题: 用户点了发布按钮后页面无响应
Why 1: 因为请求没有发出去
Why 2: 因为请求被 Promise.reject 了
Why 3: 因为 token 过期但 refreshToken 没被调用
Why 4: 因为 axios interceptor 里只处理了 401，没处理 token 过期后的 403
→ 根因: refreshToken 逻辑不完整，只覆盖了 401 状态码
```

标准：追到第 4-5 层，直到答案是一个**可以改的代码、配置或流程**。

### 第三步：分类漏测原因

从四个维度判断为什么没被拦住：

#### A. PRD 缺口
- [ ] PRD 中是否明确描述了这个行为？
- [ ] 边界条件和异常路径在 PRD 中写了？
- [ ] 验收标准覆盖了这类场景？

**常见信号：**「这个情况当时就没考虑到」「PRD 没写这个分支」

#### B. 代码缺陷
- [ ] 逻辑遗漏（if 分支、else 分支、异常处理）
- [ ] 边界条件（空列表、None、超大输入、并发冲突）
- [ ] 数据契约不一致（shared-models 字段增减未同步）
- [ ] 类型错误（运行时类型不匹配、枚举值缺失）

#### C. 测试缺口
- [ ] 这个场景有对应的测试吗？
- [ ] 如果没测：是不知道要测，还是知道但没写？
- [ ] 现有测试是否覆盖了代码路径？（行覆盖/分支覆盖）
- [ ] 测试数据是否太「干净」（只测理想输入，没测畸形输入）

**区分关键：** 不知道要测 = PRD 或架构缺口。知道没写 = 测试纪律问题。

#### D. 流程缺口
- [ ] CI 阶段是否可能发现（lint / type-check / test）？
- [ ] review 时是否可能注意到？
- [ ] 是单人疏忽还是多人都会犯的同类错误？

**区分关键：** 换个人写也会踩同样的坑 = 流程或工具问题。只有你会踩 = 个人疏忽。

### 第四步：推荐测试级别

根据问题类型，判断该补什么级别的测试。不懂技术没关系——直接看下表找到对应场景就行。

#### 测试类型速查（按成本从低到高排序）

| 测试类型 | 一句话解释 | 成本 | 发现什么问题 |
|---------|-----------|------|------------|
| **单元测试** | 测单个函数/方法。比如测一个「计算热度分」的函数输入输出对不对 | 极低 | 逻辑分支漏了、边界条件没处理、计算错误 |
| **数据契约测试** | 检查 shared-models 的定义跟实际代码用的是不是一致 | 低 | 字段名拼错、类型不匹配、新增字段没同步 |
| **状态覆盖测试** | 测组件的每一种状态：加载中、空数据、出错、成功、无权限 | 低 | 页面空白、加载转圈不停、报错信息不友好 |
| **组件/UI 测试** | 测一个页面组件渲染出来对不对，按钮能不能点 | 低 | UI 错位、按钮失效、文案错误 |
| **集成测试** | 测几个模块配合工作。比如 API 调了数据库能不能正确返回数据 | 中 | API 返回格式不对、数据查不出来 |
| **API 测试** | 直接测接口，给什么参数返回什么结果 | 中 | 接口报错、参数不合法时没正确处理 |
| **冒烟测试** | 核心功能快速过一遍——能登录吗、能发布吗、能展示吗。每次改动后跑 | 中 | 改了个小功能把核心流程搞坏了 |
| **回归测试** | 改了代码后把所有已有测试重跑一遍，确保没把原来好的搞坏 | 中 | 修了 A 把 B 搞坏了 |
| **端到端测试** | 模拟真人操作完整流程：打开页面 → 登录 → 搜索 → 点开文章 → 发布 | 高 | 复杂用户流程在某个环节断了 |
| **压力测试** | 模拟很多人同时用，看系统会不会慢或崩溃 | 高 | 页面加载慢、高并发时报错、服务器撑不住 |
| **安全测试** | 检查有没有安全漏洞，比如不登录能不能看到付费内容 | 高 | 权限绕过、数据泄露、SQL 注入 |
| **环境测试** | 在不同浏览器/手机/系统上测试 | 中 | 某些浏览器上布局乱了、功能不能用 |

#### 问题类型 → 推荐测试级别对照表

| 错误类型 | 推荐补的测试 | 为什么 |
|---------|------------|--------|
| if/else 分支漏了（某个情况没处理） | **单元测试** | 测每个分支，保证全覆盖 |
| null / 空数据 / 边界值没处理 | **单元测试** + **状态覆盖测试** | 单元测逻辑，状态覆盖测 UI 表现 |
| shared-models 字段不匹配 | **数据契约测试** | 专门查这个，别的测不到 |
| API 返回数据不对或报错 | **集成测试** + **API 测试** | 验证请求→响应的整条链路 |
| 按钮点了没反应、页面跳转不对 | **端到端测试** | 模拟真实用户操作才能发现 |
| 页面加载慢、操作卡顿 | **压力测试** | 模拟多人同时用的场景 |
| 安全漏洞（越权、数据泄露） | **安全测试** | 专项检查，别的不覆盖 |
| 页面显示错乱、样式问题 | **组件/UI 测试** + **环境测试** | 组件测试保证渲染正确，环境测试检查不同浏览器 |
| 核心功能在改完后挂了 | **冒烟测试** + **回归测试** | 冒烟快速拦截，回归全面检查 |
| 复杂流程（多步骤操作）出错 | **端到端测试** | 完整流程模拟，中间任何环节断了都能发现 |
| 并发冲突（两个人同时操作） | **压力测试** | 模拟并发场景 |
| 数据量大了就出问题 | **压力测试** | 构造大量数据测试 |

#### 选择原则

- **先选成本最低的那个**。大多数问题是单元测试或状态覆盖测试就能拦住
- **同一个问题可以补多个级别的测试**。比如边界条件漏了 → 补单元测试 + 状态覆盖测试
- **端到端测试是兜底，不是首选**。它最贵最慢，只在复杂流程出错时推荐
- **没有测试能发现需求没定义的东西**。如果根因是 PRD 缺口，补再多测试也拦不住——先补 PRD

### 第五步：制定改进措施

每项改进措施要对应到**具体文件、配置或流程变更**，不能是空话。

| 类别 | 发现 | 改进措施 | 优先级 |
|------|------|---------|--------|
| PRD 缺口 | 未定义边界条件下的行为 | 补 PRD 描述 + 验收标准 | P0 |
| 代码缺陷 | 缺少 null 检查 | 修代码 + 补单元测试 | P0 |
| 测试缺口 | 不知道有这个场景 | 补对应的测试用例 | P1 |
| 测试缺口 | 知道但没写 | 补充 TDD 纪律：必须先写失败测试 | P1 |
| 流程缺口 | 多人犯同类错误 | 加 lint 规则 / type check / CI 门禁 | P1 |
| 流程缺口 | review 看不出来 | 更新 Review Checklist 增加该检查项 | P2 |

#### 改进措施模板

```
[P<优先级>] <类别>: <措施>
  具体动作:
    - 文件/位置: <路径>
    - 改什么: <具体变更>
  同步更新:
    - PRD: 需要/不需要
    - Review Checklist: 需要/不需要
    - .clinerules: 需要/不需要
```

#### 测试建议输出格式

在改进措施中附带测试级别建议：

```
[P<优先级>] 测试补充: <建议的测试类型>
  - 测试类型: <单元测试 / 状态覆盖 / 集成 / 端到端 / ...>
  - 测什么: <具体要验证的行为>
  - 为什么这个级别: <该测试能拦截这类问题的理由>
```

### 第六步：执行与验证

1. **修代码**（如果是代码问题）
2. **补测试**（如果是测试缺口）—— 必须 RED→GREEN 流程
3. **更新文档**（PRD / Review Checklist / .clinerules / AGENTS.md）
4. **验证**：修复后确认问题不再复现

## 与 TDD + 7 阶段流程的集成

复盘不是一次性的，它应该**反向注入到开发流程中**，让下次不再出同类问题。以下是每类发现应回流到哪个阶段：

### 映射关系

```
复盘发现 ──→ 回溯到 7 阶段中的对应阶段 ──→ 产生具体改进
```

| 复盘发现 | 回溯阶段 | 对应产出 |
|---------|---------|---------|
| **PRD 缺口** | → Stage 2（PRD） | 补充验收标准、异常路径描述 |
| **架构遗漏** | → Stage 3（架构） | 补充 ARCHITECTURE.md 或方案对比 |
| **测试缺口（不知道要测）** | → Stage 4（计划） | 下次开发计划中标注该场景需测 |
| **测试缺口（知道但没测）** | → Stage 5（TDD） | RED→GREEN 流程纪律：测试先于代码 |
| **代码缺陷** | → Stage 5（TDD） | 补测试 → 修代码 → 回归验证 |
| **流程缺口（多人会犯）** | → Stage 0（AGENTS.md） | 更新 .clinerules 硬约束 |
| **流程缺口（review 漏了）** | → Stage 6（评审） | 更新 Review Checklist |

### 补测试必须走 TDD

复盘发现测试缺口后，补测试**不是**「写个测试验证一下」，而是：

```
RED    写一个能复现这个 bug 的失败测试
   ↓   测试红了才能证明它抓住了问题
GREEN  修代码让测试通过
   ↓   只做最少改动、不多修
REFACTOR  清理代码和测试
       ↓   现有回归测试全绿
```

**关键：** 先写测试看到它失败了（RED），再修代码（GREEN）。跳过 RED 直接修代码 = 你在凭运气修复，测试只是事后装饰。

### 覆盖面评估

补完测试后，评估一下这次覆盖到位没有：

| 层级 | 问题 | 评估 |
|------|------|------|
| **单点** | 只修了这个 bug | 不解决问题——下次换个地方还会犯 |
| **模式** | 同类型问题一次修完（清理所有类似的分支遗漏） | 治本——同一模式不再重复出现 |
| **流程** | 更新了 PRD / checklis / .clinerules 让这类问题以后「不可能」出现 | 系统性拦截——换个人也不会踩这个坑 |

复盘目标至少要达到**模式级**覆盖。

### 映射到 auto-exec 的 Phase

如果修复工作走 auto-exec 编排，复盘结论对应：

| 复盘改进 | auto-exec Phase |
|---------|----------------|
| 修代码 | Phase 4 执行任务 |
| 补测试 | TDD 雪球（RED→GREEN） |
| 更新 PRD | PRD 门禁（PRD-impact: true） |
| 更新 .clinerules | 写入项目硬约束 |
| 更新 Review Checklist | 下一轮 Stage 6 自动生效 |

## 完整示例

> **用户：**「我试了一下热榜筛选功能，选『科技』分类后发现返回的结果里有几个『娱乐』分类的条目混进去」

复盘:

**问题:**
- 现象: 筛选「科技」分类返回了非科技类条目
- 预期: 只返回分类=科技的文章
- 复现: /api/trending?category=tech

**根因:**
- Why 1: SQL 查询的 WHERE category=:cat 条件正常
- Why 2: 但部分文章在采集入库时 category 字段为空
- Why 3: 采集器对无法识别分类的文章留空了 category
- Why 4: 后续分类筛选时 NULL 值被某些查询忽略了
- **根因**: 采集器的分类映射不完整，未匹配的分类没有 fallback 机制

**漏测分类:**
- PRD 缺口: 是 — PRD 只说「按分类筛选」，没定义「未分类文章」的行为
- 代码缺陷: 否
- 测试缺口: 是 — 测试数据所有文章都有分类，没测 NULL category
- 流程缺口: 否

**推荐测试级别:**
- **单元测试**: 采集器的分类映射函数，验证「未匹配→fallback」逻辑
- **状态覆盖测试**: 筛选组件在 category 为空值时的表现
- **集成测试**: API 查询在 category=NULL 时的行为

**改进措施 + 阶段回流:**
- [P0] 代码: 采集器添加 fallback 分类「未分类」，入库前保证 category 非空
  → Stage 5 (TDD): RED=写一个 NULL category 测试 → GREEN=修映射逻辑 → 回归全绿
- [P1] 测试: 补充分类映射单元测试 + NULL category 集成测试
  → Stage 5 (TDD): RED 确认测试能抓到问题
- [P1] PRD: 补充「未分类文章」在筛选时的行为定义
  → Stage 2 (PRD): 补充验收标准
- [P2] 流程: Review Checklist 增加「测试数据覆盖空/边界值」检查项
  → Stage 6 (评审): 更新 Checklist

**覆盖面评估:** 模式级 ✅ — 同一"未定义边界条件"模式在所有采集器的分类映射中一次修完

```

```

## 纪律

- **问题不分大小**：即使看起来是「低级错误」也要复盘——低级错误重复出现说明机制有问题
- **不追责，追机制**：复盘是为了修系统，不是为了说「当时怎么没注意」。如果某个人错过，其他人也可能会错过
- **改进措施必须有产出**：任何「以后注意」都是空话。必须对应具体文件变更、配置更新或流程修改
- **五层追问打底**：表面原因的背后才是根因，不要停在第一层
- **测试建议要具体**：不要只说「加测试」，要说清楚加什么类型的测试、测什么行为、为什么这个级别能拦住
- **补测试必须先 RED**：跳过「写失败测试」直接修代码，测试就变成事后装饰，等于没复盘
- **目标至少是模式级覆盖**：修一个 bug 不够，要把同模式的问题全扫一遍
