# /pm — 产品需求文档助手

将模糊产品想法转化为结构化专业 PRD。

## 适用场景

- 模糊想法需梳理成文档
- 与团队同步需求
- 评估可行性前需上下文
- 产品迭代需结构化描述

## 工作流程

### Phase 1：需求澄清（5轮）

1. 目标用户：谁用？核心画像？日活？
2. 核心问题：解决什么痛点？不做会怎样？
3. 成功标准：怎么衡量？量化指标？
4. 约束条件：技术栈？时间？预算？合规？
5. 不做清单：本期不做什么

### Phase 2：方案对比

推荐方案 + 理由，至少2个方案对比。

### Phase 3：PRD输出

结构：背景→场景→规格→非功能→AC→不做清单→附录

核心要素：
- 用户场景(Happy/Alternative/Error)
- 错误状态表(空/载/错/边)
- 验收标准(可验证量化)
- 不做清单

### Phase 4：PRD审查

输出前检查：
- 目标用户明确
- 成功标准可量化
- 异常路径完整
- 不做清单完整
- 约束条件明确
- 安全/隐私已考虑

## 常见陷阱

- 跳过澄清直接写 → 猜需求
- 只写Happy Path → 异常就崩
- AC不量化 → 说不清对不对
- 没有Out of Scope → 范围蔓延
