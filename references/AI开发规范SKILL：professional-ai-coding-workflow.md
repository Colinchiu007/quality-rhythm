## 一、核心原则：让 AI 给你当「团队」，不是「打字员」

你不需要懂代码，但你需要懂**分工**。

专业开发的分工角色：
| 角色 | 职责 | 怎么跟 AI 说 |
|------|------|-------------|
| **PM（产品经理）** | 写 PRD、拆需求、定优先级 | "先帮我写一份 PRD" |
| **架构师** | 技术选型、目录结构、模块划分 | "设计一下项目架构" |
| **CTO（技术总监）** | 技术决策、代码评审、技术债管理 | "Review 一下代码质量" |
| **QA（测试）** | 写测试用例、执行测试、卡质量门禁 | "帮我跑一遍测试，没通过不能发布" |
| **COO（运营总监）** | 发布决策、用户反馈、商业判断 | "分析一下这个功能值不值得做" |

**小白最常犯的错误**：上来就说"帮我写一个 xxx 应用"——AI 直接出代码，跳过需求、设计、评审。结果是代码能用但维护不了，改一个功能全崩。

✅ **正确的做法**：让 AI 先扮演 PM，把需求写清楚；再扮演架构师，把方案画出来；最后才让开发写代码。

## 二、具体怎么做：分阶段推进

### ✅ 阶段 1：先写 PRD（产品需求文档）

跟 AI 说：
> "你作为产品经理，帮我写一份 PRD。项目是 [一句话描述]。要包含：目标用户、核心功能列表、非功能需求（性能/安全）、MVP范围。"

拿到 PRD 后审一遍，你觉得对就签字确认。**没有 PRD 不动手。**

### ✅ 阶段 2：技术架构 & 技术选型

> "现在你作为技术架构师，基于这份 PRD，帮我设计技术架构。考虑：用什么语言/框架、目录结构怎么组织、数据库设计、第三方服务选型。给出 2-3 个方案并对比优缺点。"

### ✅ 阶段 3：拆任务、写开发计划

> "你作为项目经理，把 MVP 拆成可执行的开发任务，每个任务不超过 4 小时工作量。按依赖关系排优先级，标注哪些可以并行开发。"

### ✅ 阶段 4：编码（TDD 模式）

> "你作为开发工程师，按 TDD 方式实现 [某个任务]。先写测试，再写实现代码，确保测试通过。"

### ✅ 阶段 5：代码评审 & 质量门禁

> "你作为 CTO，review 这段代码。检查：命名规范、错误处理、性能隐患、安全隐患。列出必须修复的问题和可选的优化建议。"

## 三、关键 SOP 文件模板

教 AI 记住你的开发规范，让它在每个阶段自动执行。以下是你要 AI 记住的东西：

```
# 项目开发 SOP

## 所有任务必须遵守：
1. 先写文档，再写代码
2. 代码提交前必须通过全部测试
3. 每 2-3 个功能合并后做一次 code review
4. 不允许有 warning 的代码合并到主干
5. 所有 API 关键路径必须有错误处理

## 沟通规范：
- 每次开始新任务前，先确认当前项目的目录结构和进度
- 改代码前先告诉我你要改什么、为什么改
- 遇到报错先贴完整错误栈，不要猜原因
```

你可以把这段直接粘贴到对话开头，说：
> "请记住以下 SOP，在后续所有开发中自动遵守。"

## 五、避坑清单（实操经验）

| 坑 | 后果 | 解法 |
|----|------|------|
| 直接写代码 | 后面改不动 | 先写 PRD + 架构文档 |
| 不写测试 | 加功能就崩 | 要求 TDD：测试先于代码 |

| 不建分支 | 改出 bug 无法回滚 | 用 git，feature 分支开发 |

---

## 为什么不能自动判断？

因为 Monorepo 不是一个技术对错问题，是**组织约束**：

```
Monorepo：所有项目在一个仓库
  → 优点：代码共享方便、跨项目修改一次 PR 搞定、统一版本管理
  → 缺点：仓库越来越大、CI 越来越慢、权限控制粗糙

独立仓库：每个项目一个仓库
  → 优点：项目独立部署、仓库体积小、权限精确控制、团队隔离
  → 缺点：跨项目引用要发版、依赖版本不同步、PR 分散
```

AI 不知道你们团队有几个人、项目之间耦合多深、要不要独立部署——**这些是业务决策，必须由你决定。**

```bash
# 改代码、写测试、commit
git commit -m "feat(engine): add batch optimization"
# 推送到 GitHub，创建 PR，合并到 monorepo main

# 发布引擎到 npm/pypi（从 monorepo 里发布）
cd packages/engine
npm publish  # 或 python -m build && twine upload
```

**关键：代码从来没离开过 monorepo，但每次发布都会生成独立的版本号和包。**

### 阶段 2：引擎独立运营（将来某个时点）

有两种方式"拆"：

#### 方式 A：独立仓库（最常见）

```bash
# 从 monorepo 的某个 tag 切出来
git checkout packages/engine
git checkout -b engine-standalone
# 或直接用工具切：
monorepo-split packages/engine github.com/Colinchiu007/engine
```

之后：
- `engine` 代码变成独立仓库
- 其他项目引用它：`pip install @colinchiu/engine` 或 `npm install @colinchiu/engine`
- 之前的版本、发布历史、CI/CD 全部保留

#### 方式 B：保持 Monorepo，但独立运营

```
不需要拆代码，只需要：
  - 引擎有独立的 CI/CD pipeline
  - 引擎有独立的版本发布节奏
  - 引擎有独立的监控、告警、文档
  - 其他包依赖引擎的版本固定（不自动跟随更新）
```

## AI 在 Monorepo 中的执行差异

| 场景 | Monorepo | 独立仓库 |
|------|---------|---------|
| 你说"做引擎的新功能" | AI 自动 `cd packages/engine` 操作 | AI 进入对应仓库操作 |
| commit message | `feat(engine): xxx` | `feat(xxx)` |
| 分支名 | `feat/engine-xxx` | `feat/xxx` |
| PR 范围 | 只影响 engine 包的 diff | 影响整个仓库 |
| 测试 | AI 只跑 engine 的测试 | AI 跑整个仓库的测试 |
| 发布 | AI 自动执行 `cd packages/engine && npm publish` | AI 执行 `npm publish` |
| 拆出时 | AI 执行 split 命令 | 不适用 |

**对 AI 来说，区别只是"scope"多了一个包名前缀。** 其他流程完全一样。

## 实际决策树

```
你需要拆出独立运营？
  │
  ├─ 只是"独立发版"？
  │   → Monorepo 就够了
  │   → 每个包有自己的 pyproject.toml / package.json
  │   → CI/CD 独立 pipeline
  │
  ├─ "独立团队运营"？
  │   → 中期：保持 Monorepo + 独立 pipeline
  │   → 长期：拆独立仓库（代码物理分离）
  │
  └─ "卖给别人/开源"？
      → 拆独立仓库
      → 把不相关的代码和 monorepo 彻底分开
```

## 回答你的问题

**Monorepo 完全支持"未来拆出独立运营"。** 这反而是 Monorepo 的标准能力——很多公司（Google、Facebook、Uber）都是先 Monorepo 做大，再拆成独立产品。

**AI 能自动处理这种模式吗？能。** 你只需要在架构设计阶段告诉 AI：

```
"用 Monorepo 组织所有项目，但每个子包要能独立开发和独立发布。
未来 engine 模块可能会拆成独立运营。"
```

AI 就会：
1. 按 workspace 结构创建项目
2. 每个包配置独立的 CI/CD
3. 分支/commit/PR 自动带包名前缀
4. 将来拆出时，AI 帮你执行拆分流程

**你不需要操心，AI 按规范自动执行。**

根据 `professional-ai-coding-workflow` SKILL（v1.6.0）的定义：

## 快速模式

**适用场景：**
- 小修小改
- **已知路径的实现**（比如复用已经做过的模式）
- 数据/文档型功能（不是大改架构）

**流程（3 个门禁）：**
1. PRD 更新
2. 测试（如适用）
3. 打包/文档同步

**特点：** 直接开干，**不写 PM-PRD/ARCH 文档**，但保留 3 个门禁。

## 正式流程

**适用场景：**
- 新功能（用户没做过的领域）
- 修改架构
- 多任务批量

**流程（7 阶段）：**
1. 想法澄清
2. PRD
3. 架构设计
4. 开发计划
5. 编码实现
6. 代码评审
7. 发布

**特点：** 写 PM-PRD + ARCH + TDD，每阶段 CEO 签字。

## 决策点

| 问题 | 快速模式 | 正式流程 |
|------|---------|---------|
| 用户已经做过类似的（比如这次 v0.8.0 是「Dashboard 资源展示+图片预览+模型配置」——属于前端 UI 扩展，路径已知） | ✅ | |
| 用户说"做新功能"且没做过 | | ✅ |
| 多功能批量（"都实现"） | | ✅ |
| 单个小修改（"修个 bug"） | ✅ | |
| 跨架构变更 | | ✅ |
| 需要选 2-3 方案对比 | | ✅ |

## 关键原则（来自 SKILL）

> **CEO 签字规则**：分清「确认」和「决策」。只需确认进展的（架构方案已定、测试全绿、文档已更新）→ 直接推进。需要选择的（方案 A vs B、二选一决策）→ 等 CEO 拍板。**不要每阶段都问「确认吗」**。

> 即使回复「要」这样单个字——如果这是对新任务的确认，仍然必须问用哪个模式。**单字确认模式不是跳过流程的通行证。**

## 触发确认模式的话术

```
"走正式流程（7 阶段）还是快速模式？有 X 个功能。"
```

只在以下情况问：
- 用户说"做 X"（明确新任务）
- 用户说"都实现" / "全部来" / 一系列功能
- 用户对复用提议回复"要"

---

## 快速模式 vs 正式流程

| 维度 | **快速模式** | **正式流程** |
|------|-------------|------------|
| **触发** | 已知路径、小修小改 | 新功能、改架构、批量任务 |
| **PM-PRD** | ❌ 跳过（可选 1 行计划） | ✅ 完整 |
| **ARCH** | ❌ 跳过 | ✅ 完整 |
| **TDD** | ✅ 测试必须有 | ✅ 必须 |
| **代码评审** | 可选 | ✅ 每 2-3 功能 |
| **文档同步** | ✅ 必须 | ✅ 必须 |
| **CEO 签字** | ❌ 跳过 | ✅ 每阶段 |
| **流程文档数** | 0-1 | 5+（PRD/ARCH-F1/ARCH-F2/CHANGELOG/README/AGENTS） |

## 决策树

```
用户说"做 X" / "复用 X" / "要"
    ↓
是新功能领域？  ───是──→  正式流程（7 阶段）
    ↓不是
是批量多任务？  ───是──→  正式流程（PM-PRD + ARCH-F1/2/3）
    ↓不是
是修 bug / 小改？ ───是──→  快速模式
    ↓不是
是数据/文档型？  ───是──→  快速模式（ARCH→实现→文档同步）
    ↓
    快速模式
```

## 实际判断（用过的例子）

| 任务 | 走的模式 | 原因 |
|------|---------|------|
| **v0.7.0 复用 prompt-optimizer/sd-dynamic-prompts** | 快速 | 已知路径（复用模板语法），文档同步 |
| **v0.7.0 DSL 语法** | 快速 | 路径已知 |
| **v0.8.0 资源/预览/模型配置** | 快速 | 已知路径（前端 UI 扩展） |
| **看板（F9）** | 快速 | Vue 3+Element Plus 已知栈 |
| **大改架构** | 正式 | 7 阶段必走 |
| **多功能批量（"全部实现"）** | 正式 | PM-PRD 必写 |

## 关键警告

> **不管功能多小，都走快速模式的 3 个门禁：PRD更新 + 测试（如适用）+ 文档同步。** 即使是纯数据注入，也要写 1 行 PM-PRD、更新 CHANGELOG/README/PRD。**用户介意的是没按流程走，不是文档改动大小。**

> 跳步骤用户会主动质问「为什么没按开发规划和流程走」。
