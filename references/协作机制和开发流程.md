## LLM 分级建议

核心原则：用重模型的场景是**决策点多、上下文广、错误代价高**的环节。

### 🟥 重模型（Sonnet 4 / o4-mini 级）— 花预算的地方

|环节|原因|替代方案风险|
|---|---|---|
|**架构/接口设计**|一个错误决策 = 后续所有代码白写|轻模型会漏边界条件，产出自洽但错的方案|
|**代码 Review**|需要理解设计意图 + 发现隐藏缺陷|轻模型只抓表面格式问题，放走逻辑漏洞|
|**复杂调试**|跨模块追踪根因|轻模型容易陷入局部最优，反复绕圈|
|**核心业务逻辑**（支付/鉴权/管道编排）|出错直接导致生产事故|—|
|**多 Agent 分配 / 开工决策**|你和我这一层|这个位置没有替代|
|**测试策略设计**|决定测什么、不测什么、mock 边界|轻模型会测一堆无关用例，漏关键路径|

### 🟩 轻模型（Haiku / 4o-mini / Flash 级）— 省预算的地方

|环节|前提条件|省多少|
|---|---|---|
|**CRUD 样板代码**|接口契约已锁死|~70% token|
|**单元测试实现**|测试策略已定，只写具体用例|~60%|
|**类型标注 / 格式化**|纯机械填充|~80%|
|**Config 文件**（CI yaml / Dockerfile / pyproject）|有现成模板|~70%|
|**文档 / docstring**|代码已写完，只补说明|~60%|
|**按模式的批量修改**|改法已确认，重复执行|~80%|
|**简单的 SQL / 数据转换**|逻辑清晰无歧义|~70%|

### 🟡 中庸层（哪些可上可下）

|环节|建议|判断依据|
|---|---|---|
|**写 Feature 代码**|看复杂度。接口锁死的纯实现 → 轻；含逻辑判断的 → 重|估算时判断|
|**写测试策略**|重。测什么不测什么决定了测试 ROI|—|
|**集成测试**|重。跨模块验证，漏一个场景就是漏 bug|—|
|**AI 写代码时的"自检"**|轻模型写的代码，用重模型快速过一眼|性价比最高|
