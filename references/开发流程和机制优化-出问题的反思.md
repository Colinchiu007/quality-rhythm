## 2. GitHub 搜索结论

我扫了一圈，**没有一个现成的 GitHub 工具能完整做到你说的这件事**。原因很简单：从自然语言的产品需求 → 自动枚举所有交互流程和 UI 状态，这本质上是需要 LLM 理解力的任务，不是传统静态分析工具能搞定的。

### 最接近的几个方向

|方案|做法|局限|
|---|---|---|
|**Storybook CSF + test-runner**|用 Component Story Format 定义每个状态的 story，test-runner 自动转成测试|需要人工手动定义每个状态，不会从需求自动推导|
|**XState 状态机**|把 UI 建模为形式化状态机，从 statechart 自动推导可达状态和测试路径|建模成本高，而且需要人工把需求翻译成状态机|
|**Playwright Codegen**|录制用户操作生成测试代码|只能覆盖正常路径，不会自动想到空态/错误态|
|**AI 生成测试（CodiumAI / Diffblue）**|从代码上下文自动补测试|只能基于现有代码，不是从产品需求出发|
|**Swagger/OpenAPI → 契约测试**|从 API 定义生成请求/响应验证|只有接口层面的，没有前端 UI 状态|

### 真正能解决你需求的方案

你想要的这个能力——**「给一个产品需求，自动枚举所有交互流程和非正常状态 UI 文字」**——最适合的实现方式就是**一个专门的 AI Agent Skill**，也就是我上面那个 skill 的 Phase 1 部分。

事实上，这个 skill 已经内置了那个逻辑。当你激活 `/page-level-tdd-testing` 并给出一个需求时，它会：

1. **Step 1**：梳理所有用户交互路径（正常路径 A/B/C 各自怎么走）
2. **Step 2**：生成状态覆盖矩阵（每个数据源的 Loading / Empty / Error / Success / Edge Case）
3. **Step 3**：枚举所有非正常状态的 UI 文字，逐条映射到测试用例
4. **Step 4**：按 TDD 逐个实现

要不要现在就试一下？给一个产品需求，用 `/page-level-tdd-testing` 跑一遍流程，看看它能不能枚举出你想要的那些东西。

系统会根据 skill 的 `description` 字段来判断是否主动触发。我保存时写的描述是：

所以当你的需求**自然落在这些关键词范围内**时，系统会自动建议激活。比如你说：

- "帮我给这个新页面写测试"
- "新做了一个表单页，先枚举一下所有 UI 状态再写测试"
- "分析一下这个功能的所有交互流程和异常提示文字"

但自然语言匹配不是 100% 可靠的——描述越精确匹配率越高。如果你觉得匹配不够敏感，我可以**更新 description 补上更多触发词**，比如加上「枚举」「状态覆盖」「交互流程」「非正常状态」「UI 提示」这些。
