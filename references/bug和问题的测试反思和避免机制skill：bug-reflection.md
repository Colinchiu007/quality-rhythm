做一个新skill，目的是我每次自己测试项目功能或使用中发现一个问题，你就要反思一下这个问题或bug产生的原因和为什么没有测试出来，是PRD写得不清晰，缺明确的描述，还是代码写的有问题，或是测试机制不到位，然后进行相应的机制和流程的优化，避免下次发生这类问题。

---

已保存为 `/bug-reflection`，所有对话可用。

结构很简单：**描述问题 → 5 Whys 找根因 → 四维分类(PRD/代码/测试/流程) → 具体改进措施 → 执行验证**。

核心设计理念：

- **不追责，追机制**——换个人也会踩同样的坑才是真问题
- **改进措施必须有产出**——废除「以后注意」这类空话，必须对应到具体文件变更
- **结果写入 memory**——同类问题可回溯，避免重复分析
---
## 使用场景

任何你**手动测试或日常使用**时发现的问题：

|场景|例子|
|---|---|
|页面行为异常|点了按钮没反应、报错、跳转不对|
|数据不对|筛选结果混入不相关条目、排序错误|
|逻辑遗漏|某个分支一直没处理、边界条件挂了|
|体验问题|但体验问题不触发复盘，只触发普通优化|

**不适用：** CI 自动发现的、纯样式问题、已有明确根因的 typo。

## 触发方式

两种：

```
/bug-reflection
```

直接发，skill 会引导你把问题描述出来，然后自动走复盘流程。

或者在对话中口述问题，然后说"复盘一下"——我会认出来走这个流程。

---

## 大概流程（你实际体验到的）

```
你: /bug-reflection
我: 请描述你遇到的问题——发生了什么，预期应该是什么？

你: 热榜筛选"科技"分类，结果里有几条娱乐的混进来了
我: 开始复盘 →
    ① 问题确认：分类筛选返回了不匹配的条目
    ② 5 Whys：追问根因
    ③ 漏测分类：PRD没定义"未分类"的筛选行为 + 测试数据太干净
    ④ 改进措施：[P0] 采集器加 fallback 分类 / [P1] 补测试 / [P1] 补 PRD
    ⑤ 执行：修代码 → 补测试 → 同步 PRD → 写入 memory
```

整个流程你只需要第一步描述问题，剩下我来做。
